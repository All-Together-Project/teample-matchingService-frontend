# AllTogether — AI 검색 · 추천 시스템 설명서

이 문서는 본 프로젝트에 도입된 두 가지 AI 기능 — **AI 자연어 검색**과 **AI 맞춤 추천** — 이 실제로 어떻게 작동하는지, 일반인도 이해할 수 있도록 비유와 그림 중심으로 설명한다.

---

## 1. 왜 AI를 도입했나

기존의 키워드 검색만으로는 사용자의 의도를 정확히 파악하기 어렵다.

| 사용자 입력 | 키워드 검색 결과 | 실제 사용자가 원한 것 |
|---|---|---|
| "정처기 같이 준비할 사람" | "정처기"가 들어간 글만 매칭 | 정보처리기사, 자격증, CS 스터디까지 |
| "주말에 친구 만들고 싶어요" | "친구"가 들어간 글만 매칭 | 친목 모임, 동호회, 네트워킹 행사 |
| "디자인 사이드 프로젝트" | "디자인"+"사이드" 모두 있는 글 | UI/UX, 브랜딩, Figma 활용 프로젝트 |

→ **단어가 아니라 "의미"로 검색해야 한다**는 결론. 이를 위해 두 가지 AI 기능을 만들었다.

1. **AI 자연어 검색** — 검색어의 의미를 이해해서 비슷한 의미의 게시글을 찾아준다.
2. **AI 맞춤 추천** — 사용자가 자유롭게 입력한 요청에 맞춰 어울리는 프로젝트나 팀원을 추천한다.

---

## 2. 핵심 개념: "의미"를 숫자로 표현하기 (임베딩)

AI 검색의 핵심은 **임베딩(Embedding)** 이라는 기술이다. 어려운 개념이지만 비유로 풀면 단순하다.

### 비유: "도서관 지도"

도서관의 모든 책을 거대한 지도 위에 배치한다고 상상해보자.

- 비슷한 주제의 책은 **서로 가까운 자리**에 둔다.
- 다른 주제의 책은 **멀리 떨어진 자리**에 둔다.
- 예를 들어 "토익 문법책"과 "TOEIC 단어집"은 옆자리, "캠핑 가이드"는 반대편 끝.

이때 각 책의 위치를 **숫자 좌표(x, y)** 로 표현할 수 있다. 위치가 비슷하다 = 좌표가 가깝다 = 의미가 비슷하다.

실제로는 (x, y) 같은 2차원이 아니라 **768차원의 좌표**를 사용한다. 사람은 768차원을 시각화할 수 없지만, 컴퓨터는 768개의 숫자 묶음으로 의미를 매우 정밀하게 표현할 수 있다.

### 좌표를 만드는 방법

좌표는 사람이 손으로 매기지 않는다. **AI 모델(Google Gemini)** 이 자동으로 부여한다.

```
"토익 900+ 스터디 모집"  →  AI 임베딩 모델  →  [0.12, -0.45, 0.78, ..., 0.03]  (숫자 768개)
"외국인 친구 사귀는 모임" →  AI 임베딩 모델  →  [0.55,  0.21, -0.13, ..., 0.91]
"러닝 크루 모집합니다"    →  AI 임베딩 모델  →  [-0.34, 0.67, 0.42, ..., -0.18]
```

이 모델은 인터넷의 수십억 개 문장을 학습해서 "어떤 문장끼리 의미가 비슷한지" 알고 있다. 그래서 같은 의미라면 단어가 달라도 비슷한 좌표를 만들어낸다.

### 비슷함을 측정하는 방법: 코사인 유사도

두 좌표가 얼마나 가까운지 계산하는 공식이 **코사인 유사도(Cosine Similarity)** 다.

- 두 벡터가 같은 방향을 가리키면 **1.0** (완전히 비슷함)
- 직각이면 **0.0** (관련 없음)
- 정반대면 **-1.0** (정반대 의미)

검색어와 게시글들의 벡터를 모두 비교해서 **유사도가 높은 순으로 정렬**하면 검색 결과가 된다.

## 3. AI 자연어 검색 작동 흐름

사용자가 검색창에 "토익"이라고 입력하고 **AI 검색** 버튼을 눌렀을 때 일어나는 일을 단계별로 설명한다.

### 전체 흐름도

```
[사용자]
   │  "토익" 입력 → AI 검색 클릭
   ▼
[웹 프론트엔드]
   │  postApi.semanticSearch({ query: "토익" })
   ▼
[Supabase Edge Function: semantic-search]
   │
   │  ① 검색어를 Gemini로 임베딩
   │     "토익" → [0.12, -0.45, ..., 0.03]  (768개 숫자)
   │
   │  ② 데이터베이스에서 모든 게시글의 좌표와 거리 계산
   │     SQL: SELECT (1 - 코사인거리(검색벡터, 게시글벡터)) AS 유사도
   │          FROM posts
   │          WHERE 유사도 >= 0
   │          ORDER BY 유사도 내림차순
   │          LIMIT 12
   │
   │  ③ 결과 게시글에 작성자/태그 정보 붙임
   │
   ▼
[웹 프론트엔드]
   │  유사도 높은 12개 게시글 카드 렌더링
   ▼
[사용자]
   결과 확인 (각 카드에 일치도 % 표시)
```

### 각 단계 상세 설명

**① 검색어 임베딩**
- 사용자가 입력한 자연어를 Google Gemini의 임베딩 모델(`gemini-embedding-001`)이 768차원 벡터로 변환한다.
- 검색용 모드(`RETRIEVAL_QUERY`)를 사용 — 게시글을 등록할 때 쓰는 모드(`RETRIEVAL_DOCUMENT`)와 다르다. 검색어와 문서를 서로 다르게 최적화하면 매칭 정확도가 올라간다.
- 이 호출은 0.3초 정도 걸린다.

**② 유사도 계산 (pgvector)**
- 데이터베이스(PostgreSQL)에 **pgvector**라는 확장 기능을 설치해두었다. pgvector는 벡터 간 거리 계산을 매우 빠르게 처리한다.
- 게시글마다 미리 임베딩 벡터가 저장돼 있으므로, 검색어 벡터와의 거리를 일괄 계산해서 정렬한다.
- 보통 0.05초 안에 수백 개 게시글을 비교한다.

**③ 결과 보강**
- 유사도가 가장 높은 12개 게시글을 선정.
- 각 게시글의 작성자 이름·매너 온도·태그도 함께 가져와서 카드 형태로 만든다.

**총 응답시간**: 약 0.5초.

### 게시글 등록 시: 임베딩은 언제 만들어지나?

게시글이 새로 작성되거나 수정될 때마다 별도의 Edge Function(`embed-post`)이 호출되어 그 게시글의 임베딩을 만들고 데이터베이스에 저장한다. 즉, 검색 시점이 아니라 **등록 시점에 미리 좌표를 매겨둔다**. 그래야 검색이 빠르다.

임베딩 input에는 다음이 모두 포함된다:
```
[제목] React 입문 스터디
[카테고리] STUDY / 코딩/개발
[태그] React, 프론트엔드, 초보환영, 온라인
[본문] ...
```

→ 카테고리와 태그까지 함께 임베딩하므로, 본문이 비슷하더라도 분야가 다르면 다른 좌표가 부여된다.

## 4. AI 맞춤 추천 시스템

추천 시스템에는 **두 가지 모드**가 있다.

### 4-1. 프로젝트 추천 (사용자 → 모집 공고)

사용자가 마이페이지에서 자유롭게 입력한다. 예:
> "주말에 만나서 자격증 같이 준비할 사람들이랑 모이고 싶어요"

이 요청을 AI가 분석해서 가장 어울리는 모집 공고를 찾아준다.

**작동 흐름:**

```
[리더 자유 입력]
   │  "주말에 자격증 준비 모임"
   ▼
[ai-recommend Edge Function]
   │
   │  ① 요청문을 임베딩 (검색 모드)
   │
   │  ② 게시글들과의 유사도 계산 → 상위 8개
   │
   │  ③ AI에게 "왜 이 게시글이 추천되는지" 한 줄 이유 생성 요청
   │     모델: gemini-2.5-flash-lite (JSON mode)
   │     8개를 한 번에 묶어서 호출 (효율적)
   │
   ▼
[결과 8개 — 각 게시글에 추천 이유 첨부]
   예: "정처기 자격증 + 주말 진행 일정 일치"
       "자격증 준비 동기와 일치, 매너 온도 높음"
```

**핵심 차이점 (검색 vs 추천)**

| 항목 | AI 검색 | AI 추천 |
|---|---|---|
| 사용자 입력 | 키워드/짧은 문장 | 자유로운 자연어 요청 |
| AI 호출 | 임베딩만 | 임베딩 + 추천 이유 생성 |
| 결과에 표시 | 일치도(%) | 일치도 + 추천 이유 한 줄 |
| 응답 시간 | ~0.5초 | ~2초 (이유 생성 때문) |

### 4-2. 팀원 추천 (모집 공고 작성자 → 후보 사용자)

게시글을 올린 사람(리더)이 자기 모집 공고에 적합한 팀원을 찾고 싶을 때 사용한다. "팀원 추천 받기" 버튼을 누르고 자유롭게 요청 입력:
> "React 경험 있고 빠르게 작업할 수 있는 사람"

**작동 흐름:**

```
[게시글 작성자(리더)]
   │  "React 경험자 빠른 작업 가능자"
   ▼
[ai-recommend Edge Function — member 모드]
   │
   │  ① 해당 게시글의 태그 조회 (예: React, 프론트엔드, 사이드프로젝트)
   │
   │  ② 그 태그들을 가진 사용자 목록 수집
   │     - 한 사용자가 여러 태그를 공유할수록 점수↑
   │     - 작성자 본인과 기존 멤버는 제외
   │
   │  ③ 후보가 8명보다 적으면 매너 온도 높은 사용자로 보충
   │
   │  ④ 후보 정렬: 태그 겹침 수 > 매너 온도 순
   │
   │  ⑤ 상위 8명 각각에 대해 AI 추천 이유 생성
   │     "자기소개에 React 경험 언급, 매너온도 38.2도로 신뢰성↑"
   │
   ▼
[리더 화면]
   8명의 후보 카드 (닉네임, 매너온도, 태그, 추천 이유)
   → 마음에 드는 후보에게 쪽지로 직접 컨택
```

### 4-3. AI 없이 작동하는 기본 추천 (Phase 1)

AI를 호출하지 않고도 작동하는 **태그 기반 추천**이 따로 있다. 마이페이지에 들어가면 자동으로 노출되는 "당신에게 추천하는 모집"이 이것이다.

```sql
-- 핵심 로직 (의사 SQL)
SELECT 게시글
FROM 모집 중인 게시글
LEFT JOIN 사용자_관심태그 ON 사용자.태그 = 게시글.태그
ORDER BY 겹치는 태그 개수 DESC, 최신순
LIMIT 6
```

- 비용 0원 (AI 호출 없음)
- 즉시 응답
- 단점: "왜 추천되는지" 설명이 없고, 의미 추론을 못 함 (정확히 같은 태그만 매칭)

→ 따라서 **무거운 AI 추천은 사용자가 명시적으로 요청할 때만** 호출하고, 평소엔 가벼운 태그 매칭으로 처리하는 이원화 구조를 채택했다.

## 5. 사용한 기술 스택

### Google Gemini API

구글이 제공하는 생성형 AI. 본 프로젝트에서는 두 가지 모델을 용도별로 사용:

| 모델 | 용도 | 호출 빈도 |
|---|---|---|
| `gemini-embedding-001` | 텍스트를 768차원 좌표로 변환 (임베딩) | 게시글 등록 시, 검색 시 |
| `gemini-2.5-flash-lite` | 짧은 텍스트 생성 (추천 이유) | AI 추천 시에만 |

임베딩 모델은 매우 빠르고 저렴해서 사실상 무제한 호출 가능. 텍스트 생성 모델은 호출량이 많아지면 비용이 발생하므로 추천 이유 생성처럼 꼭 필요한 경우에만 사용.

### pgvector (PostgreSQL 확장)

벡터 검색을 데이터베이스 내에서 직접 수행하게 해주는 확장 기능. 별도의 벡터 데이터베이스를 구축할 필요가 없어 운영 부담이 적다.

- 768차원 벡터를 저장하는 컬럼 타입(`vector(768)`) 제공
- 코사인 유사도, 유클리드 거리 등 다양한 거리 함수 지원
- **IVFFlat 인덱스**로 수천~수만 건의 게시글에서도 0.1초 안에 검색

### Supabase Edge Functions

Deno 런타임 기반의 서버리스 함수. 본 프로젝트에서 3개 함수가 작동:

| 함수 | 트리거 | 역할 |
|---|---|---|
| `embed-post` | 게시글 작성/수정 시 자동 | 임베딩 생성 → DB 저장 |
| `semantic-search` | 사용자가 AI 검색 클릭 | 검색어 임베딩 → 유사 게시글 반환 |
| `ai-recommend` | 사용자가 AI 추천 클릭 | 임베딩 + 추천 이유 생성 |

서버를 따로 관리하지 않아도 되고, 호출량에 따라 자동으로 확장된다.

## 6. 기존 키워드 검색과의 차이점 (실제 예시)

### 예시 1: "토익" 검색

| 검색 방식 | 결과 |
|---|---|
| **키워드 검색** | "토익"이라는 단어가 들어간 게시글만 매칭 (3개) |
| **AI 검색** | TOEIC, 영어 회화, 어학 시험 준비, 외국어 교류까지 의미적으로 가까운 게시글 모두 (10여 개), 유사도 순 정렬 |

### 예시 2: "주말에 운동하는 모임"

| 검색 방식 | 결과 |
|---|---|
| **키워드 검색** | "주말"+"운동"+"모임" 모두 들어가야 매칭 (거의 없음) |
| **AI 검색** | 러닝 크루, 등산 모임, 풋살 동호회, 클라이밍 모임 등 — "운동"이라는 단어가 없어도 의미가 같으면 매칭 |

### 예시 3: 사이드 프로젝트 추천

리더 요청: "주말에만 진행 가능한, React 잘 아는 디자이너랑 팀 짜고 싶어요"

- **키워드 매칭만 사용**: "React"+"디자이너"+"주말"이 정확히 들어간 사람 검색 → 거의 없음
- **AI 추천**: 자기소개에 "프론트엔드 부캐", "Figma + 코드 둘 다 만짐", "본업 직장인" 같은 표현을 가진 후보를 의미적으로 매칭 + "본업이 있어서 평일 빠듯, 주말 작업 선호" 같은 추천 이유 자동 생성

## 7. 향후 발전 방향

### 단기 (Phase 3 보완)
- **하이브리드 검색**: 키워드 점수와 의미 점수를 가중 합산해서 정확도 향상
- **사용자 행동 학습**: 사용자가 클릭하거나 지원한 게시글의 임베딩을 누적해서 "취향 벡터" 생성
- **추천 다양성**: 비슷한 게시글만 뜨지 않게 결과에 다양성 제약 추가 (MMR 등)

### 중기
- **개인화 임베딩 미세조정**: 우리 서비스 내 사용자 행동 데이터로 임베딩 모델 fine-tuning
- **부정적 피드백 반영**: "이 추천 별로예요" 버튼 → 비슷한 게시글 가중치 자동 하향

### 장기
- **대화형 매칭**: 챗봇이 사용자와 대화하며 요구사항을 정교화 → 점진적으로 후보를 좁혀가는 방식
- **이미지 임베딩**: 포트폴리오 이미지나 첨부 파일까지 의미적으로 분석

## 8. 자주 묻는 질문 (FAQ)

**Q1. AI 검색을 켜면 어떤 차이가 있나요?**
A. 일반 검색은 단어가 정확히 맞아야 결과가 나오지만, AI 검색은 단어가 달라도 의미가 비슷하면 결과에 나옵니다. 예를 들어 "정처기"라고 검색해도 "정보처리기사" 게시글이 함께 나옵니다.

**Q2. "일치도 %"는 어떻게 계산되나요?**
A. 검색어와 게시글의 의미 벡터 사이의 코사인 유사도(0~1 사이)에 100을 곱한 값입니다. 100%에 가까울수록 의미가 거의 같다고 판단된 것이고, 0%에 가까우면 거의 관련 없습니다.

**Q3. AI 추천이 느린 이유가 뭔가요?**
A. 검색은 임베딩 한 번만 호출하지만, 추천은 임베딩 + 추천 이유 생성까지 두 번의 AI 호출이 필요합니다. 그래서 약 2초 정도 걸립니다.

**Q4. 매번 AI를 호출하면 비용이 비싸지 않나요?**
A. 임베딩 모델은 호출 비용이 매우 저렴하고, 게시글 등록 시에만 한 번 호출되므로 부담이 거의 없습니다. 추천 이유 생성은 사용자가 명시적으로 요청할 때만 발생하므로 통제 가능합니다.

**Q5. 결과가 가끔 이상한 이유는?**
A. AI는 게시글 본문의 내용을 학습해서 의미를 추측합니다. 본문이 너무 짧거나 정형화된 문장으로만 이루어져 있으면 게시글들의 임베딩이 비슷해져서 변별력이 떨어질 수 있습니다. 이를 보완하기 위해 본문뿐 아니라 카테고리·태그까지 함께 임베딩 input에 포함시켜 변별력을 회복시켰습니다.

### 한 줄 요약
> **사람의 말을 컴퓨터가 이해할 수 있는 좌표로 바꾸고, 비슷한 좌표를 가진 게시글·사용자를 찾아주는 시스템**

게시글 등록 시 미리 좌표(임베딩)를 매겨두고, 검색 시점에는 검색어의 좌표만 새로 계산해서 가장 가까운 게시글을 빠르게 찾아낸다. 여기에 짧은 추천 이유 한 줄을 AI가 자동 생성해 붙여주면 "왜 이게 추천되는지" 설명되는 매칭 시스템이 완성된다.
